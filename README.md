# SellerPilot AI — AI-директор для Ozon + Wildberries

Сделано:

- единый интерфейс для кабинетов **Ozon** и **Wildberries**;
- подключение магазинов по реальным API-ключам;
- backend-прокси, который делает реальные обращения в API маркетплейсов;
- синхронизация товаров, цен и остатков там, где права токена позволяют;
- отправка обновления остатков в Ozon/WB;
- AI-рекомендации по проблемам: нет остатков, малые остатки, неактивные карточки, кросс-листинг Ozon ↔ WB;
- опциональное подключение OpenAI-compatible LLM через переменные окружения.

## Как запустить

Нужен Node.js 18+.

```bash
node server.js
```

Откройте лендинг:

```text
http://localhost:8787
```

Рабочий кабинет:

```text
http://localhost:8787/app.html
```

В интерфейсе нажмите **«Подключить Ozon/WB»** и внесите ключи.

## Где брать ключи

### Wildberries

ЛК WB → Настройки → Доступ к API. Для полноценной работы нужны права/категории API:

- Content / карточки товаров;
- Statistics / остатки;
- Prices & Discounts / цены;
- Marketplace / изменение остатков, если хотите отправлять остатки из кабинета.

Для обновления остатков WB дополнительно нужен `Warehouse ID`.

### Ozon

ЛК Ozon Seller → Настройки → API-ключи:

- `Client-Id`;
- `Api-Key`.

## AI-рекомендации

По умолчанию работает локальный rule-based анализ без внешних AI-сервисов.

Чтобы подключить LLM, задайте переменные окружения:

```bash
AI_API_KEY=ваш_ключ AI_MODEL=gpt-4o-mini node server.js
```

Для другого OpenAI-compatible провайдера:

```bash
AI_API_KEY=ваш_ключ AI_BASE_URL=https://your-provider.example/v1 AI_MODEL=your-model node server.js
```

## Важно про безопасность

`server.js` хранит ключи локально в `data/shops.json`. Для продакшена нужно заменить это на БД/секрет-хранилище и добавить полноценную авторизацию пользователей.

Не отправляйте API-ключи маркетплейсов напрямую из браузера: это небезопасно и часто блокируется CORS.


## Деплой на Vercel (демо для тестов)

Проект адаптирован под Vercel serverless:

- `api/index.js` — точка входа serverless-функции, оборачивает тот же обработчик, что и `server.js`;
- `vercel.json` — все запросы `/api/*` идут в функцию, статика (`index.html`, `app.html`, `admin.html`) отдаётся напрямую;
- на Vercel данные пишутся в `/tmp/sellerpilot-data` (на проде файловая система только для чтения).

Деплой через CLI:

```bash
npm i -g vercel
vercel login
vercel        # превью-деплой
vercel --prod # боевой деплой
```

Или через сайт: импортировать репозиторий на vercel.com → Deploy (настройки не нужны, `vercel.json` всё описывает).

После деплоя:

- открыть выданный домен — это лендинг;
- `/app.html` — рабочий кабинет;
- поле **Backend API URL** оставить пустым (фронт и бэкенд на одном домене).

Как пользоваться демо:

1. На странице входа нажать **«⚡ Заполнить тестовый аккаунт»** и зарегистрироваться (или войти).
2. **«Подключить WB»** → вставить свой реальный WB token → пойдёт реальная синхронизация через WB Seller API.
3. Без токена можно смотреть встроенные demo-магазины и demo-товары.

> ⚠️ Важно про демо-хранилище: на Vercel `/tmp` — **эфемерное**. Пока инстанс «тёплый», всё работает: регистрация, токены, синхронизация. После простоя/холодного старта данные (пользователи, магазины, товары) сбрасываются — для тестового демо это нормально. Реальные вызовы WB/Ozon API при этом всегда живые. Для постоянного хранения нужно подключить БД (Vercel KV / Postgres / Upstash).

Telegram-планировщик (`setInterval`) на serverless не запускается — он работает только при локальном `node server.js`. Остальные Telegram-эндпоинты (`/api/telegram/test` и т.д.) работают и на Vercel.

## Регистрация пользователей

В `app.html` теперь есть настоящие формы:

- регистрация пользователя по email + паролю;
- вход по email + паролю;
- logout;
- токен сессии хранится в `localStorage` браузера;
- backend хранит пользователей в `data/users.json`, сессии в `data/sessions.json`;
- пароль хранится не открытым текстом, а в виде PBKDF2-хэша;
- магазины и товары фильтруются по пользователю, у каждого аккаунта свои кабинеты Ozon/WB.

Для продакшена нужно заменить JSON-файлы на БД и добавить восстановление пароля / email-подтверждение.

## Восстановление доступа через администратора

Восстановление пароля сделано вручную через администратора:

- пользователь нажимает **«Восстановить»** на странице входа;
- указывает email/телефон и данные для подтверждения личности;
- заявка сохраняется в `data/recovery_requests.json`;
- в интерфейсе показан телефон администратора: **89382222453**;
- администратор после проверки личности вручную помогает восстановить доступ.

API заявки:

```text
POST /api/auth/recovery-request
```

## Новые AI/операционные модули

Добавлены backend-модули, которые работают поверх реальных Seller API Ozon/WB:

- синхронизация заказов/продаж за последние 30 дней;
- `GET /api/dashboard` — Health Score, оценка потерь, AI-задачи, план поставок, утечки прибыли;
- `GET /api/orders` — единая лента заказов WB/Ozon;
- `POST /api/costs` — загрузка себестоимости по SKU для расчёта маржи;
- AI-рекомендации теперь используют не только остатки, но и продажи, цену, себестоимость и статус карточки.

Используемые реальные API:

- WB: карточки, остатки, цены, orders/sales из Statistics API;
- Ozon: product list, product info, stocks, FBS/FBO postings, import stocks.

Часть API зависит от прав токена. Если токен не имеет нужной категории, сервис сохранит предупреждение в магазине, но не уронит всю синхронизацию.

## Потери за последние 7 дней + прогноз

AI-dashboard теперь считает ущерб именно по темпу продаж за последние 7 дней:

- `estimatedLoss7d` — прогноз потерь на 7 дней;
- `estimatedLoss14d` — прогноз на 14 дней;
- `estimatedLoss30d` — прогноз на 30 дней;
- `lostOrders7d` — примерное количество потерянных заказов за 7 дней;
- `revenue.sevenDays` — выручка за 7 дней;
- `counters.orders7d` — количество заказов за 7 дней.

Если за 7 дней данных мало, backend берёт fallback по 14/30 дням, чтобы рекомендации не были пустыми.

## Telegram-бот AI-директора

Добавлена ежедневная отправка отчёта в Telegram по времени пользователя.

API:

```text
GET  /api/telegram/settings
POST /api/telegram/settings
POST /api/telegram/test
```

Пользователь указывает:

- Telegram Bot Token;
- Chat ID;
- время отчёта, например `09:00`;
- часовой пояс, например `Europe/Moscow`;
- включить/выключить ежедневный отчёт.

Backend раз в минуту проверяет расписание и отправляет AI-отчёт через официальный Telegram Bot API:

```text
https://api.telegram.org/bot<TOKEN>/sendMessage
```

Токен хранится в `data/telegram_settings.json`, в интерфейсе показывается только маска. Для продакшена нужно хранить токен в зашифрованном виде/secret storage.

Как получить chat_id:

1. Создать бота через `@BotFather`.
2. Написать боту любое сообщение.
3. Открыть `https://api.telegram.org/bot<TOKEN>/getUpdates`.
4. Взять значение `message.chat.id`.

## Gemini только для дорогих текстовых задач

Gemini подключается опционально и используется **только** для:

- AI-чата;
- красивого weekly-отчёта;
- генерации карточек;
- ответов на отзывы.

Расчёты потерь, Health Score, неликвид, ABC/XYZ, закупки и маржа остаются rule-based и не тратят токены.

Запуск с Gemini:

```bash
AI_PROVIDER=gemini \
GEMINI_API_KEY=ваш_ключ \
GEMINI_MODEL=gemini-1.5-flash \
AI_MAX_CALLS_PER_DAY=30 \
node server.js
```

Проверить статус:

```text
GET /api/ai/status
```

Gemini API endpoints в проекте:

```text
POST /api/chat
GET  /api/report/weekly
POST /api/ai/card
POST /api/ai/review-reply
```

Если `GEMINI_API_KEY` не задан, все эти функции работают в бесплатном fallback-режиме `rules`.

## Gemini 2.5 Flash для AI-анализа бизнес-модулей

По умолчанию модель Gemini теперь:

```text
gemini-2.5-flash
```

Gemini может анализировать не только чат/отчёты/карточки/отзывы, но и бизнес-модули:

- потери за 7 дней;
- Health Score;
- неликвид;
- ABC/XYZ;
- план закупки;
- unit-экономика;
- убыточные SKU;
- безопасная цена;
- AI-задачи.

Важно: backend всё равно сначала считает точные цифры формулами, а Gemini 2.5 Flash делает слой AI-директора: объясняет, приоритизирует и формирует управленческие выводы. Это снижает риск галлюцинаций и экономит токены.

Запуск:

```bash
AI_PROVIDER=gemini \
GEMINI_API_KEY=ваш_ключ \
GEMINI_MODEL=gemini-2.5-flash \
AI_MAX_CALLS_PER_DAY=30 \
node server.js
```

Gemini-анализ возвращается в:

```text
GET /api/intelligence
GET /api/dashboard?gemini=1
```

## Ручная оплата через WhatsApp + админ-кабинет

Оплата Start 1 000 ₽/мес теперь работает вручную через WhatsApp.

Поток:

1. Пользователь в кабинете нажимает **Оплатить 1 000 ₽/мес**.
2. Backend создаёт invoice в `data/billing.json`.
3. Пользователя перекидывает в WhatsApp на номер администратора: **89382222453**.
4. В сообщении уже есть логин и комментарий платежа.
5. Админ открывает отдельный сайт:

```text
http://localhost:8787/admin.html
```

6. Админ входит по паролю `ADMIN_PANEL_PASSWORD`.
7. Видит invoice и пользователя.
8. Нажимает **Выдать 30 дней**.
9. Доступ пользователя становится активным.

Переменные:

```bash
ADMIN_PANEL_PASSWORD=сложный_пароль
ADMIN_PHONE=89382222453
ADMIN_WHATSAPP=79382222453
```

Если `ADMIN_WHATSAPP` не задан, номер `8938...` автоматически превращается в `7938...` для ссылки `wa.me`.

Админ API:

```text
POST /api/admin/login
GET  /api/admin/overview
POST /api/admin/activate
POST /api/admin/deactivate
```

В `admin.html` все ответы API выводятся в блок **Console / API response** и дополнительно пишутся в DevTools console через `console.log('[ADMIN]', ...)`.
